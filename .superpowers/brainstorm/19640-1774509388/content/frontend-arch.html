What should the dashboard be built on?

This is the key architectural decision — it determines UI quality ceiling, packaging, and performance

A

Stay Streamlit — polish what's there

Enhance the existing 4-tab Streamlit app with better layout, custom CSS, and the FastAPI live server already embedded as an iframe for real-time data. Fast to improve, familiar codebase.

Pros

  • No rewrite needed
  • All existing logic preserved
  • Fast to iterate

Cons

  • 1 Hz polling ceiling
  • Hard to package as .exe
  • Limited UI control
B

Hybrid — Streamlit config + FastAPI live view (recommended)

Keep Streamlit for the analysis, experiment setup, and agentic pipeline tabs. Fully replace the Live Sensor tab with the FastAPI+uPlot server already in the codebase, extended with controls. One Python process, two ports.

Pros

  • Real-time at true 20 Hz
  • Professional live UI
  • Builds on existing live_server.py
  • Moderate effort

Cons

  • Two UI paradigms in one app
  • Still hard to package as .exe
C

Full Web App — FastAPI backend + HTML/JS frontend

Replace Streamlit entirely. FastAPI backend serves all data via REST + WebSocket. All tabs rewritten as HTML/CSS/JS pages with uPlot charts. Packaged as a Docker container or a Python-bundled server you open in a browser.

Pros

  • True instrument-grade UI
  • 20 Hz across all views
  • Easily packaged/deployed
  • Highest quality ceiling

Cons

  • Largest effort — full rewrite
  • Lose Streamlit's rapid prototyping

My recommendation: B — the live server foundation is already there, and it gets you 80% of the quality at 30% of the effort
